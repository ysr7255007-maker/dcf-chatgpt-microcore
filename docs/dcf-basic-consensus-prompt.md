# DCF 基本共识提示词

本文件维护 DCF 项目的基础共识，用于新会话恢复方向感。它不是长篇治理文档，也不是详细架构说明。

DCF 是一个个人 Tampermonkey 脚本，不是企业插件平台或社区插件生态。GitHub 仓库公开只是为了更新与分发，产品语义上仍是用户与 AI 共同维护的私人模块库。

DCF 的运行模型是声明式包、本地解释器与确定性构建。包是不可变输入，用户状态独立保存，运行 registry 是可以随时重建的派生缓存。

安装、更新、停用、卸载和版本回退都只改变当前输入集合，然后走同一条候选构建与原子提交路径。包不得直接修改旧 registry，也不使用反向补丁还原历史结果。

模块定义与用户结果必须分开。模块可以提供改变外观、设置和内容的能力，但用户操作产生的值属于用户状态；模块停用或卸载不反向抹除用户结果。

包资源使用稳定地址。不同包声明同一独占资源时应在候选构建阶段明确失败，不得依靠安装顺序或隐式 `deepMerge` 决定胜负。替换核心资源必须显式声明。

样式保持来源分层。每个包拥有自己的样式片段，运行时 CSS 只是派生拼接结果；卸载包直接移除其来源片段。appearance CSS 不得重新拥有壳体几何，壳体实际边界由内核运行时围栏统一保证。

风险不由版本更新或热更新天然决定。安全关键是：候选结果先完整构建并验证，成功后再提交；应用前有恢复点，失败后旧运行状态保持不变。

外观设定不属于 userscript 本体。userscript 只保留最小挂载、解释器、构建器、宿主桥、诊断、恢复和无法避免的应急可见性能力；具体外观、模块、Surface、内容和显示策略由包或用户状态提供。

版本更新只补当前内核无法通过声明式包与本地构建完成的通用能力。功能、UI、配置、内容、样式、模块组织和显示策略应优先作为包定义；用户偏好和操作结果写入用户状态。

模块不是 UI 单元。模块是能力包。Surface 才是插件自己的显示单元。工作区、维护区和弹药区只是 area、group、order、display 元数据，不应被固定槽位绑死。

解决复杂问题时先做源头检查。不要围绕旧结构枚举分支和堆补丁；优先改变事实模型、状态边界或统一约束，使大部分复杂性失去产生条件。

维护输出应尽量短，先推进真实可用状态。除非底座能力边界或基本共识变化，不要用长文档替代实现。
