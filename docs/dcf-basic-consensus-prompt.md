# DCF 基本共识提示词

本文件维护 DCF 项目的基础共识，用于新会话恢复方向感。它不是长篇治理文档，也不是详细架构说明。

DCF 是一个个人 Tampermonkey 脚本，不是企业插件平台。它的目标是作为 ChatGPT 页面内的轻量能力总线、热更新承载层和个人工作台。

默认规则：功能、UI、配置、内容、样式、模块组织、Surface 布局都应优先通过热更新完成。

允许改 userscript 本体的情况只保留少数底座例外：启动链路、运行模型边界、宿主桥、通用解释器或渲染原语、存储迁移、诊断链路。

运行模型边界来自旧实现失败经验：DCF 不回到远程代码、字符串执行、分片加载那套路线。这不是把某个语法写成安全表演，而是确认 DCF 的运行模型是声明式包 + 本地解释器。

判断 UI 需求时，不问“这个要不要改 DOM”。只问“现有热更新 manifest 能不能声明”。能声明就只发热更新包；不能声明时，userscript 只补通用解释器或渲染原语，不能把当前用户偏好写进默认值。

模块不是 UI 单元。模块是能力包。Surface 才是插件自己的显示单元。工作区、维护区、弹药区都只是 Surface 的 area、group、order、display 元数据，不应被固定槽位绑死。

源码发布和偏好应用必须分离：源码发布补通用能力，热更新包应用具体偏好。当前用户喜欢的顶栏、固定底部、隐藏滚动条、箭头按钮、宽度等，都属于热更新偏好。

维护输出应尽量短，先推进真实可用状态。除非底座变化或共识变化，不要用长文档替代实现。
