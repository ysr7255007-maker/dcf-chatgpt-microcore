# DCF 基本共识提示词

本文件维护 DCF 项目的基础共识，用于新会话恢复方向感。它不是长篇治理文档，也不是详细架构说明。

DCF 是一个个人 Tampermonkey 脚本，不是企业插件平台。它的目标是作为 ChatGPT 页面内的轻量能力总线、热更新承载层和个人工作台。

DCF 的热更新面应采用黑名单模型，而不是白名单模型。默认不是“只有列出来的字段能改”，而是“除明确禁止的边界外，声明式配置都应能改”。随着需求增加，不应不断增加特例白名单，而应保持少数清晰禁止项，并让 manifest / registry 承载开放的声明式元数据。

外观设定不应属于 userscript 本体。userscript 可以保留最小挂载、解释器、宿主桥和无法避免的应急可见性样式，但不应把具体外观、布局偏好、颜色、标签显示、按钮组合、卡片样式等作为本体默认设计。外观应作为声明式 appearance / theme / layout manifest 进入 registry，并由热更新或用户配置应用。

默认规则：功能、UI、配置、内容、样式、模块组织、Surface 布局和内容库行为都应优先通过热更新完成。热更新包可以提供新的声明式数据、元数据、显示策略、动作配置、内容类型配置、Surface 配置和样式 token；userscript 负责解释这些数据，而不是把每一种偏好写成源码特例。

允许改 userscript 本体的情况只保留少数底座例外：启动链路、运行模型边界、宿主桥、通用解释器或渲染原语、存储迁移、诊断链路。

运行模型边界来自旧实现失败经验：DCF 不回到旧的动态逻辑路线。DCF 的运行模型是声明式包 + 本地解释器。

判断 UI 或内容库需求时，不问“这个字段是否已经在白名单里”。先问“这是否只是声明式元数据或偏好”。只要没有触碰黑名单边界，就应让热更新承载。只有现有解释器完全不能解释某类通用声明时，userscript 才补通用解释能力，而且不能把当前用户偏好写进默认值。

模块不是 UI 单元。模块是能力包。Surface 才是插件自己的显示单元。工作区、维护区、弹药区都只是 Surface 的 area、group、order、display 元数据，不应被固定槽位绑死。

源码发布和偏好应用必须分离：源码发布补通用能力，热更新包应用具体偏好。当前用户喜欢的顶栏、固定底部、隐藏滚动条、箭头按钮、宽度、标签显示方式等，都属于热更新偏好。

热更新包里，改变渲染结构或行为的 UI 偏好属于 ui_policy；纯尺寸、间距、宽度等 CSS token 属于 styles。不能把 structural UI 字段全塞进 styles 后假定所有内核都会解释。

维护输出应尽量短，先推进真实可用状态。除非底座变化或共识变化，不要用长文档替代实现。
