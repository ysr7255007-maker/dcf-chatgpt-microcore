# DCF 基本共识提示词

本文件维护 DCF 项目的基础共识，用于新会话恢复方向感。它不是长篇治理文档，也不是详细架构说明。

DCF 是一个个人 Tampermonkey 脚本，不是企业插件平台。它的目标是作为 ChatGPT 页面内的轻量能力总线、热更新承载层和个人工作台。

DCF 的更新模型采用黑名单模型，而不是白名单模型。默认不是“只有列出来的字段能改”，而是“除明确禁止的边界外，声明式配置都应能改”。随着需求增加，不应不断增加特例白名单，而应保持少数清晰禁止项，并让 manifest / registry 承载开放的声明式元数据。

风险不由“版本更新”或“热更新”天然决定。把变化写进 userscript 并不会自动安全；把变化放进热更新也不会自动危险。安全的关键是：更新前有快照，更新中有事务，更新后可验证，失败后可回滚。

外观设定不属于 userscript 本体。userscript 可以保留最小挂载、解释器、宿主桥和无法避免的应急可见性样式，但不应把具体外观、布局偏好、颜色、标签显示、按钮组合、卡片样式等作为本体默认设计。外观应作为声明式 appearance / theme / layout manifest 进入 registry，并由热更新或用户配置应用。

版本更新不是治理边界，也不是偏好边界。只有当前内核在能力上无法接收、保存、解释、执行或恢复某类更新时，才允许 userscript 版本更新。其余变化都应做成热更新或冷更新：热更新直接改 registry 并立即生效；冷更新允许需要刷新、重新扫描、重新挂载或重建视图，但不应要求安装新版 userscript。

默认规则：功能、UI、配置、内容、样式、模块组织、Surface 布局、内容库行为、显示策略、标题、排序、分组、标签显示、按钮组合和主题外观，都应优先通过冷热更新完成。userscript 负责解释这些数据，而不是拥有这些设计。

架构应围绕“可回滚更新管线”搭建：更新包先进入待应用区，应用前保存 registry 快照和变更摘要，应用时生成事务记录，应用后运行最小诊断。若页面无法工作、渲染异常、诊断失败或用户不满意，应能从 UI、维护请求或本地存储恢复到上一快照。

必须版本更新的范围应只剩能力不可达项：启动前元信息、脚本最小加载链路、当前内核无法自我安装的解释器能力、宿主权限或浏览器权限缺失、无法由现有诊断恢复的损坏、以及必须由真实代码实现的新本地能力。只要当前内核已有承载与解释能力，就不能把该变化升级为版本更新。

运行模型边界来自旧实现失败经验：DCF 不回到旧的动态逻辑路线。DCF 的运行模型是声明式包 + 本地解释器。

判断任何需求时，不问“这个字段是否已经在白名单里”，也不问“这是不是外观/DOM”。先问“当前内核是否在能力上无法通过冷热更新做到”。若不是能力不可达，就应让 manifest / registry 承载。若确实能力不可达，userscript 也只补通用承载或解释能力，不能把当前偏好写进默认值。

模块不是 UI 单元。模块是能力包。Surface 才是插件自己的显示单元。工作区、维护区、弹药区都只是 Surface 的 area、group、order、display 元数据，不应被固定槽位绑死。

源码发布和偏好应用必须分离：源码发布补当前内核无法做到的能力，热更新或冷更新应用具体偏好。

维护输出应尽量短，先推进真实可用状态。除非底座能力边界或基本共识变化，不要用长文档替代实现。
