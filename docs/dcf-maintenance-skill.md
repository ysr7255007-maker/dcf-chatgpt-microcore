# DCF 维护技能

本文件维护 DCF 的操作技能，用于约束实际改动步骤。它应保持短小、可执行，不承担长篇架构治理。

处理任何 DCF 需求时，先做分流：

1. 默认采用冷热更新，不采用版本更新。热更新直接写 registry 并立即生效；冷更新允许需要刷新、重新扫描、重新挂载或重建视图，但不安装新版 userscript。
2. 只有当前内核在能力上无法接收、保存、解释、执行或恢复某类更新时，才允许 userscript 版本更新。
3. 现有冷热更新表达不了的，先判断是不是解释器过度白名单化。优先开放声明式承载面，而不是为每个新需求新增一个特例字段。
4. userscript 只补当前内核能力不可达的通用承载、解释、宿主桥、恢复或本地能力，不把当前偏好写进默认行为。
5. 补完能力后，用热更新或冷更新应用用户偏好。

安全不由更新方式决定。版本更新不会天然安全，热更新不会天然危险。维护时把重点放在可回滚：应用前快照、应用中事务、应用后诊断、失败后恢复。

外观设定不应放在 userscript 本体。userscript 只保留最小可见、可恢复的应急样式和通用解释器；正式外观、布局、标签显示、颜色、按钮组合、卡片样式、密度、间距等，都应放在 appearance / theme / layout manifest 或 registry 配置中。

必须版本更新只看能力不可达，不看需求类型。典型能力不可达包括：启动前元信息必须变化、脚本最小加载链路损坏、当前内核无法安装或解释新的声明类型、宿主权限或浏览器权限缺失、现有诊断恢复链路失效、必须由真实本地代码实现的新能力。除此之外，功能、UI、外观、内容、动作、排序、分组、标题、标签、样式和显示策略都应走冷热更新。

热更新黑名单包括：旧的动态逻辑路线、破坏启动链路、破坏宿主桥、破坏存储迁移、破坏诊断链路、无确认删除或危险操作、把当前偏好写进 userscript 默认值。

黑名单之外的声明式数据应默认允许进入 registry。字段、元数据、显示策略、动作配置、内容类型配置、Surface 配置、样式 token、标题、排序、分组、area 标记、标签显示规则，都不应因为没有被逐项列入白名单而被拒绝。

任何更新包应用前都应能形成一个恢复点。恢复点至少保存 registry 快照、应用包摘要、时间、来源 hash、影响范围和前后差异摘要。危险操作应先生成可撤销记录，再改变当前状态。

维护面板应提供恢复能力：查看最近恢复点、回滚上一恢复点、回滚指定恢复点、导出当前 registry、导入 registry、清除坏包影响。回滚优先恢复 registry，不要求安装旧 userscript。

可调节 UI 不能用少量枚举值假装手动控制。壳体宽高、贴顶贴底、避让距离等连续性调节，应优先做成通用 appearance 调节能力和步进命令；按钮应执行增量调整或锚点切换，而不是套用几个固定预设 CSS 包。

壳体不得依靠每个按钮、锚点或距离选项分别推导可视区上限。可视区安全应由内核统一读取实际 `visualViewport` 和壳体 `getBoundingClientRect()`，通过一个与命令来源无关的运行时围栏限制最终可见矩形；registry 继续保存用户期望几何。

模块内部执行自己的维护调节时，不应自动给用户发送 DCF_FEEDBACK。反馈用于对话回传和诊断，不是每次本地按钮动作的副作用。命令可显示 notice、建立恢复点、更新 registry；只有明确需要回传时才发送 feedback。

禁止把“需要渲染真实按钮”“需要改变外壳结构”“需要改 DOM”“这是 UI 功能”“这需要新参数”“这有安全风险”当作版本更新理由。真正理由只能是：当前内核在能力上无法通过冷热更新完成，或者当前内核没有能力可靠回滚这类变化。

改源码前必须能说清楚新增的是哪个能力面，以及为什么当前内核不能通过冷热更新做到。说明必须落到能力不可达或回滚不可达，而不是落到偏好、字段名、UI 形态或抽象安全感。

内容库动作应优先由 content_types.actions 等热更新配置决定。删除类动作要远离常用动作，优先作为角标或独立危险区，并保留确认。

更新类动作应生成“基于当前对话更新原资产”的请求，提供原内容、当前话题/新洞见的更新意图，并要求输出完整可重新摄取的内容块；若输入框非空，应复制请求而不是覆盖用户草稿。

看到 DCF_FEEDBACK 后先看 kernel_version。若用户仍在旧内核，新的配置可能只是被保存而不会立即生效；需要说明哪些能力当前内核已有，哪些能力确实需要版本更新。

当按钮存在但点击无效、部分生效或视觉结果与 registry 不一致时，先获取该次点击的相关联命令轨迹，包括命令解析、逐步能力调用、调用前后 registry appearance、持久化状态与壳体计算样式。未取得执行输入、返回值和可观察结果前，不继续替换模块或猜测内核原因。若当前内核无法提供这些证据，缺失的通用诊断链本身属于能力不可达，应先恢复诊断链。

“有日志”不等于“有证据”。有效命令证据必须以一次点击的 `trace_id` 关联完整链路，包含模块身份、命令解析、逐步能力调用、变更前状态、内存变更、持久化结果、渲染后结果和最终命令状态。外观问题还必须包含命中的 CSS 规则来源，不能只看 registry 或 computed style 的单点值。

证据链不得改变被观察对象的行为。日志序列化、存储或回传失败时，真实命令仍应继续执行；诊断应降级到当前标签页的内存记录并显式报告降级。证据按当前标签页和当前内核启动隔离，不混入旧版本或其他标签页记录。

证据默认做隐私过滤，不持久化或回传对话正文、提示词、内容资产、令牌、认证信息、Cookie、密码和完整 registry 快照。页面中出现维护请求标记时只进入待授权状态；只有用户在本地开启一次维护回传后，才允许发送受限诊断字段。反馈生成、复制和成功送达必须区分记录。

输出安装块时要注意 DCF 会扫描可见文本。只有确实要让当前页面安装时，才输出完整安装块标记。说明示例时拆开标记或改用文字描述。

每次仓库改动后至少做这些检查：确认版本号、确认 meta 与 user 一致、检索是否回到旧失败路线关键词、运行仓库内 `npm test`、拉取关键行引用。若创建或更新共识，应同步更新本提示词文档或技能文档。
