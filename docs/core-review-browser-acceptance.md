# DCF Core Review 浏览器验收单

## 隔离

验收时只启用 Core Review，暂时停用完整捆绑审查版。完整捆绑版保留作为回滚基线。

## 已获得的现场证据

`0.1.0-alpha.1` 已经证明：

- 空 Core 可以显示最低恢复入口；
- 官方插件包可以导入并持久化；
- 精确快照可以建立；
- 12 个代码单元已经写入本机代码库；
- 首个失败发生在插件代码执行前，而不是包、存储、哈希或快照阶段。

失败原因是 Core Review 未声明沙箱需求，Tampermonkey 将其优先注入页面上下文，动态执行继承了 ChatGPT CSP。`0.1.0-alpha.2` 固定使用 `@sandbox DOM`，并在启动插件之前执行一次运行环境自检。

## A. 空 Core

1. 安装或更新到 Core Review `0.1.0-alpha.2`。
2. 打开新的 ChatGPT 页面。
3. 左下角应出现 Core Review 入口。
4. 若旧实验状态仍存在，可保留现有代码库和快照直接继续；无需重复导入。

## B. 执行环境复验

1. 在恢复面板中清除强制恢复，重新尝试当前 minimal 快照。
2. 若隔离沙箱可用，Shell 应开始启动。
3. 若环境仍不允许执行本机代码单元，应直接显示 `dynamic_execution_unavailable`，且 `boot.error.stage` 为 `dynamic_execution`；不得再误报为 `dcf.next.shell` 插件故障。
4. 记录完整状态和浏览器控制台错误。

## C. 加载 minimal

1. 页面刷新后应出现 DCF Shell。
2. 应启动页面交互、插件管理器和维护诊断。
3. Core Review 面板中 boot 应为 completed，最近可用快照应等于当前快照。

## D. 已安装与当前运行分离

1. 打开插件面板。
2. 将语言弹药、外观和备份加入当前组合并重启。
3. 刷新后这些插件应启动；未加入的插件不应自动运行。

## E. 插件运行环境

1. 修改外观后刷新，确认设置保留。
2. 新建测试弹药后刷新，确认内容保留。
3. 维护诊断中的已启动插件和页面状态应与当前组合一致。

## F. 本机 Agent 插件

1. 通过插件面板把本机 Agent 加入当前组合。
2. 本机服务未运行时，插件应显示未连接，但 Core 不应进入恢复。
3. 完成一次 Echo 往返，确认结果返回原页面。

## G. 恢复

基础步骤通过后，再使用专门的故障插件版本验证启动失败、最近可用快照与最低恢复入口。

每一步记录页面现象、面板状态、控制台错误、当前快照 ID 和 boot.plugins。不能以“看起来正常”代替状态证据。
