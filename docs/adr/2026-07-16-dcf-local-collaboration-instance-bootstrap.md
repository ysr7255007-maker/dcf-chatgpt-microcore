# ADR: DCF 本地协作实例与本机 Agent 自举

Date: 2026-07-16  
Status: accepted

## Context

DCF Next 已经具备最小生存盒、普通插件、ChatGPT 页面交互、持久化、诊断、备份和确定性 Action 发布。后续讨论中出现了文件交付、GitHub 上传、网页定位、现场诊断、自动升级和跨平台迁移等需求。若分别实现，它们会在 userscript 中形成多套运输、状态和恢复机制。

这些需求共同依赖的根能力并不是某一种上传协议，而是：当前网页 DCF 实例能够主动连接一个本机执行端，把任务交给真实工作区中的工具型 Agent，并把结果送回发起任务的同一页面实例。

## Decision

1. 不重构 DCF Next，不建立新的协作 Core、服务总线、命令解释器或多 Agent 调度层。
2. 新能力实现为普通内嵌插件 `dcf.next.local-agent`；“本地协作实例”只是该插件与本地桥共同维护的运行绑定，不进入生存盒业务状态。
3. 当前网页实例主动向 loopback Bridge 注册。注册连接本身承担页面定位；第一版不反向枚举浏览器窗口、标签页、屏幕坐标或 CDP Target。
4. Bridge 只监听 loopback，使用短期配对码和进程期会话令牌。GitHub、模型、SSH 与 Agent 凭据不进入 DCF 存储或备份。
5. 网页只提交工作区别名和 `dcf.local-task.v1` 自然语言任务。Bridge 不暴露任意 Shell HTTP 端点；真实路径只存在本机配置中。
6. Bridge 只负责绑定页面、映射工作区、调用一个已配置的工具型 Agent、保存任务状态和返回结果。文件修改、测试、Git 与 GitHub 操作由现有 Agent、`git` 和 `gh` 完成。
7. DCF 复用现有 Shell 面板、ChatGPT 完整回复观察和输入框写入，不复制页面选择器、MutationObserver 或第二套 UI。
8. 包裹体只作为控制消息边界。检测到任务后必须由用户点击确认；第一版结果只填入输入框，不自动发送。
9. 本地 Bridge 离线属于插件普通状态 `disconnected`，不能令插件启动抛错或触发生存盒恢复。
10. PR 与 GitHub Action 继续承担正式差异观察、独立验证和确定性生成物发布；本地 Agent 不取代该阶段。
11. 修正启动清单演化：保存清单表示用户对当前插件集合的顺序、版本和启用偏好。当前构建新增插件进入当前清单；旧最近可用组合中新增插件先以禁用状态出现，保证失败时能够回退。
12. 第一版只接一个 Agent 命令入口。只有第二个真实宿主出现后，才考虑适配器化；不提前建设通用 Agent 平台。

## First bootstrap acceptance

首个完整验收必须由网页端任务驱动本机 Agent 对 DCF 仓库完成一个低风险真实修改，运行 `verify:next`，原子提交并创建草稿 PR，等待 Action，再把结果回填到原网页输入框。新版安装后应重新注册并报告新版本。

## Consequences

- DCF Next 的既有成果被直接复用，本次主要增加一个普通插件和一个独立本地小进程。
- 文件交付、GitHub、诊断和升级不再各自扩张 userscript；它们可在连接成立后作为本机 Agent 的任务或后续运输适配自然加入。
- 当前页面与本机现场通过主动注册建立确切关系，不依赖脆弱的窗口猜测。
- Bridge 或 Agent 故障不会影响 DCF 的普通网页功能；现有最近可用组合、安全模式、旧版和 Tampermonkey 历史继续承担恢复。

## Reconsideration conditions

- 两个以上真实 Agent 宿主反复需要不同调用协议；
- 真实二进制工件传输无法在当前 HTTP 任务接口上自然扩展；
- 多个网页平台需要共享稳定的页面实例协议；
- loopback Bridge 无法满足所需浏览器外部诊断，且 CDP/扩展能力已成为真实阻塞。
