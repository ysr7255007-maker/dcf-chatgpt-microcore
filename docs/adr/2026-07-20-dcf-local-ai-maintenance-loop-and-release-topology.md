# ADR: DCF 本地 AI 维护循环与发布拓扑

- 日期：2026-07-20
- 状态：已接受；作为长期默认工作流程
- 范围：DCF 日常修改、故障修复与真实浏览器验收的开发与发布流程

## 背景

DCF 是个人维护的认知基础设施，单一用户与 AI 协作开发。此前的修改流程依赖 PR 闸门与人工审查，但存在两个根本问题：

1. **运行时黑箱**：AI（远端 ChatGPT 与本地 AI）都无法直接观测浏览器里 DCF 的真实运行状态，调试只能依赖用户口头描述加代码推断，随着项目复杂度上升越来越困难。
2. **PR 闸门阻塞自主迭代**：BrowserClaw 使本地 AI 能够自主完成「改—重载—测—取证—再改」闭环后，PR 的人工审查闸门会让这个闭环退化为半人工，失去数量级的迭代速度优势。

需要一套既快（无 PR 闸门）、又安全（只有验收过的改动到达用户）、且可审计（修改记录与回滚点持久可见）的流程。

## 决策

### 1. 单一活动开发线，直推不建 PR

日常维护只维护一条活动开发线（当前为 `rebuild/chrome-native-host-v2`）。不建功能分支，不创建 PR，不等待合并。每轮有效修改直接形成提交并推送到活动开发线。

### 2. 一轮修改一个原子提交

一次提交表达一个完整、可验收、可撤销的变化（如「修复启用并添加卡死」「修复诊断假健康」）。不按文件拆分提交，不把多个未形成完整价值的中间步骤连续推入仓库。允许本地 AI 在本地工作区反复试验；对 GitHub 暴露的是这一轮形成的完整候选。

### 3. 先提交再真实验收；运行事实高于源码推断

GitHub 提交不是「已完成」的证明，而是 BrowserClaw 测试循环的输入。源码正确、构建通过、脚本已注册都不能单独证明 DCF 在当前页面正常。本地 AI 必须通过 BrowserClaw 在真实页面取得运行证据（DCF Shell 是否存在、ShadowRoot 与面板是否挂载、点击是否返回、状态是否真实变化、标签是否真实加入、是否有控制台或底座错误、诊断是否与页面事实一致）。

### 4. Git 负责可逆，不负责阻断；GitHub 是事实来源

风险管理依靠：清晰的原子提交、当前已知良好提交、完整 Git 历史、后续修复提交、必要时 `git revert`。不依靠 PR 在修改进入仓库前阻断。

修改记录与回滚点**必须 push 到 GitHub**，本地 commit 不够——其他 AI session、用户换设备、任何审查者都看不到本地 commit。省掉的是 PR 审查闸门，不是 GitHub 本身。GitHub 仍是代码与修改记录的唯一事实来源（符合 README 原则 1）。

### 5. 两个发布状态：活动 HEAD + stable 发布指针

一条开发历史，外加一个 `stable` 发布指针：

```text
活动开发 HEAD：A → B → C → D → E
stable 指针：            ↑ C（验收通过后快进到 E）
```

- `stable` 不是独立开发线，不在其上写代码，只指向最近一次通过真实验收的提交。
- 实现为 tag 或 fast-forward-only 分支均可；承重约束是「只在验收通过后移动」。
- 活动 HEAD 自由前进（含未验收实验）；用户日常环境只消费 `stable`。
- 未验收的 push 不会立即更新用户正在使用的 DCF。

### 6. 两个浏览器环境的精确边界

用户零手动切换。两个环境都由系统管理：

- **日常环境**：用户正在使用的 DCF，永远跟随 `stable`。
- **维护环境**：BrowserClaw 操控的试验台，加载当前待测 commit；用户从不进入。

关键区分——**标签隔离 ≠ 扩展/Profile 隔离**：

- BrowserClaw 的标签所有权是真实的：本地 AI 只能操作自己开的标签，用户标签的内容受保护，页面级工具（snapshot/act/navigate/close）会拒绝对用户标签派发。
- 但 DCF 扩展状态是整个 Profile 共享的，不是按标签独立。底座重载、插件源更新会波及该 Profile 所有标签里的活 DCF。标签所有权拦不住这类操作，因为动的不是用户的标签，而是用户标签里也在跑的扩展。

由此得出执行边界：

| 改动类型 | 执行环境 | 方式 |
|---|---|---|
| 插件改动（绝大多数） | 主浏览器 | BrowserClaw 开自己的标签测页面行为，不重载扩展，标签隔离足够 |
| 底座改动（很少） | 主浏览器 + 错峰 | 本地 AI 预告并确认后重载底座，用户暂离；底座重载可恢复 |

底座重载是「可恢复的打扰」而非数据丢失：持久化状态（`chrome.storage` 里的 plugin_data、current/candidate/LKG 快照、弹药、工作区）在扩展重载后存活，只有开放标签里的运行时（Shell/user scripts）需刷新重新注入。

**独立测试 Profile 从硬前提降级为可选的未来全自动化**，当前不是启动闭环的阻塞条件。底座不能在同一 Profile 同时运行开发版与商店稳定版（扩展 ID、存储、Service Worker、User Scripts 来源会冲突）；若未来要全自动化底座测试，再建独立 Profile。

### 7. 晋升与回滚自动化

验收通过后由系统执行晋升：

```json
{ "commit": "abc123", "acceptance": "passed", "target": "<验收目标>", "evidence": "<维护运行记录>" }
```

1. 确认真实页面结果通过；
2. 确认相关快速测试通过；
3. 将 `stable` 指针快进到该 commit；
4. 更新插件稳定索引；
5. 记录上一稳定 SHA；
6. 通知用户「新的稳定检查点已晋升」。

回滚：普通插件可自动退回 Last Known Good；`stable` 可退回上一检查点；底座发布可恢复上一个商店版本或回滚构建。**发布失败只回滚发布目标，不回滚源码。**

### 8. 用户介入边界

本地 AI 自主完成技术循环。用户是产品中控台，只在体验判断无法自动决定时介入一次：

- 外观是否更顺眼、按钮位置是否符合习惯、语言弹药交互是否自然、某行为虽无报错但是否符合真正需求。

不应要求用户介入：手工安装候选、重载扩展、刷新页面、点击固定测试动作、复制诊断包、搬运控制台错误、报告提交 SHA/标签页 ID/扩展版本、在网页 AI 与本地 AI 之间转发中间结果。

技术验收通过后，用户只做一次产品体验判断，同意即自动晋升 `stable`。用户可随时中止维护循环。

### 9. 发布与开发分离

活动开发线允许快速直接提交，正式发布基于已知良好检查点：

- 普通功能插件走 GitHub 插件源；
- Chrome 底座走非公开 Chrome Web Store；
- 未通过真实验收的提交不得进入稳定发布。

## 与现有原则的关系

- **README 原则 1（GitHub 是事实来源）**：加强而非违反。所有改动仍 push，修改记录与回滚点持久可见；变化只是审查从「PR 上人工看」前移到「循环内自动取证」，证据写进 commit 与维护记录。
- **README 原则 10（不合并 PR，ChatGPT 审查、用户批准）**：演化为「`stable` 晋升由 BrowserClaw 真实验收 gating，体验判断由用户批准」。

## 未实现 / 后续

- 独立测试 Profile：可选的未来全自动化，当前非硬前提。
- 确定性脚本化：构建、哈希、版本同步、提交组装、重载、目标选择、证据采集、晋升应由确定性脚本或明确接口完成，本地 AI 不靠自然语言猜测确定性状态（待落地）。
- Local Agent 后台持续控制（pause/resume 等）：非 DCF 主线，不在本流程范围。

## 第一阶段验收

第一阶段只要求跑通一个完整维护闭环，不先建设通用自治平台。完成标志不是写完一套框架，而是本地 AI 能在不要求用户搬运信息的情况下，自行完成至少两轮：

```text
修改 → Push → 重载 → 真实测试 → 发现仍有问题 → 再修改 → 再 Push → 再测试 → 得到可确认结果
```

首个目标故障：外观插件点击「启用并添加」后不得无限停在「正在启用」，最终必须明确成功、失败或需要刷新，且诊断能反映当前页面事实。
