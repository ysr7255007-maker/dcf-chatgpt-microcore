# ADR: DCF 插件包通过编译快照进入 Tampermonkey

Date: 2026-07-17  
Status: accepted

## Context

Core Review 已在真实 Chrome、ChatGPT 与 Tampermonkey 环境中完成两轮现场验证。官方插件包能够导入，代码单元能够持久化，SHA-256 校验与精确运行快照能够建立，启动失败也能被生存核准确记录。

唯一失败前提是从持久化文本直接执行插件模块。默认页面上下文与明确请求的 DOM 沙箱都受到当前页面 CSP 约束，插件尚未启动就被拒绝。继续增加页面注入、Blob、Data URL 或多沙箱 fallback，会把一个环境限制转化为长期复杂加载器。

## Decision

1. 结束“从本机存储直接执行插件源码”的运行时路线，Core Review 实验以获得确定限制证据结束。
2. 官方插件包、本机代码单元描述、插件版本和推荐快照继续保留；它们已经证明是有效的源码与交付边界。
3. 精确运行快照从运行时对象改为构建输入：先选择插件，再把最小生存核和所选插件确定性物化为完整 userscript。
4. JavaScript 的实际加载、解析和执行交回 Tampermonkey。DCF 不再实现第二套代码解释或注入机制。
5. 生成 minimal、standard、complete 三条官方快照更新通道。三个产物使用同一 userscript 身份，用户安装另一条通道时是在替换当前代码组合，而不是并行运行多个 DCF。
6. 完整捆绑审查版继续保留为兼容回退；官方快照通过真实浏览器验收后，才决定正式默认通道。
7. 朁来的任意自定义组合由插件管理器导出意图，再由本机 Agent 或确定性构建器生成 userscript；不把任意组合运行能力压回生存核。

## Consequences

- 生存核继续只加载构建时已经确定的一组模块，不理解远程插件包、权限、依赖或代码执行策略。
- 插件包仍可独立演化；插件更新只改变快照中选中的插件源码和版本，虽然 Tampermonkey 最终安装的是完整 userscript。
- 安装、更新、切换组合和回滚统一为“安装另一份明确快照”。
- 当前静态构建成果得到直接吸收，不需要重写十个插件或改变 `start(context)` 契约。
- Core Review 生成物保留为实验记录，但不再是候选运行架构，也不进入默认验证与发布链。
