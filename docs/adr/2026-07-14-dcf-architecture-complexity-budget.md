# ADR: DCF 架构复杂度预算与简化审计

Date: 2026-07-14  
Status: proposed

## Context

DCF 最近的重构在局部上持续统一状态、事务、资源、页面和 Runtime 观察，但小修改的实际扩散面与完成时间没有下降。0.11.4 到 0.18.2 之间，系统新增了 Environment Intent、Environment Reconciler、资源图、声明式 View、Profile、模块替代、性能控制与归因、底座/能力包同步和有状态命令反馈等机制。部分机制解决了真实旁路与重复状态，另一些机制则用于修补前述分层产生的新协调缺口。

当前风险是形成复杂度补偿循环：新增抽象带来更多状态与同步边界，随后再增加迁移、诊断、替代、同步和文档机制保障这些边界。局部实现因此更统一，但整体修改成本、理解成本和失败面持续上升。

## Proposed decision

- 暂停新增通用架构层、通用协议和平台化能力，除非现有两个以上真实能力已经产生不可接受的重复旁路。
- 架构优劣以用户摩擦、边际修改成本、概念数量、状态数量、失败模式和可删除代码为主要证据，不以抽象命名、层次完整或未来可扩展性单独证明。
- 对现有机制执行简化审计：每个 Intent、Resolver、Reconciler、Resource Compiler、View Projection、Profile、Catalog、supersession、health/receipt 与迁移机制都必须指出当前真实使用者、移除后损失和最小替代方式。
- 保留已经证明降低真实风险的底线：单一权威状态、用户数据与默认值分离、确定性构建、有界 Host Adapter、隐私边界和可恢复提交。
- 普通 UI、文案和局部行为默认走最小修复；不因“可以更通用”自动升级成 Core、协议或完整架构发版。
- 新抽象必须附带删除计划，并证明它减少的长期分支、状态或旁路多于新增的协调面。
- 在简化审计完成前，不把跨平台、多作者包生态或高度通用的对话环境平台作为当前实现的默认规模假设。

## Evidence to collect

- 各版本的手写源码、生成物、测试、ADR 和维护文档增长；
- 典型局部修改触及的文件、层次、测试和发布步骤；
- 当前每个架构对象的真实消费者数量；
- 由架构分层自身产生的缺口，例如底座与能力包不同步、迁移期模块重复和投影/现场偏差；
- 删除或合并机制后的功能与恢复风险。

## Reconsideration condition

当 DCF 已有多个独立能力作者、多个真实平台实现或多个不能用较简单结构承载的产品族时，可以重新评估平台化抽象。重新引入时仍需用实际减少的重复与协调成本证明其价值。
