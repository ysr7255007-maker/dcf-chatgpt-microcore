# ADR: 先完成完整首版，再统一进行浏览器验收

Date: 2026-07-15  
Status: accepted

## Context

最小生存盒骨架完成后，原计划把“真实浏览器确认正常壳体和安全模式可见”作为进入语言弹药闭环前的阶段门。用户指出，这只是一个规模有限的 userscript 插件；在功能尚不完整时单独验收抽象骨架，既难理解，也不会显著降低后续排障成本。

## Decision

- 取消“骨架浏览器验收通过后才继续开发”的阶段门。
- 按完整产品闭环直接实现首版：语言弹药、ChatGPT 页面交互、插件管理、数据迁移与备份、长对话减负、问答性能归因、外观和维护诊断。
- 保持正式 0.18.2 不变，所有新版工作继续位于独立 `src-next/`、独立 userscript 和独立重构分支。
- 开发期间使用模块级单元测试、确定性构建和生成物语法检查发现结构问题。
- 完整首版形成后，再进行一次可理解的真实浏览器验收；发现问题时按拥有该功能的插件局部修正，不回到旧式全系统协调。

## Consequences

- 用户验收对象从抽象生存盒变成可实际使用的完整 DCF Next。
- 浏览器验收不再阻塞代码完成，但仍是正式切换前不可省略的证据。
- 生存盒、插件和数据边界必须通过自动测试保持可恢复性，避免把所有风险留到最后一次人工验收。

## Reconsideration condition

只有当后续某项能力需要不可逆数据变更、外部付费服务、系统权限或会破坏正式 0.18.2 时，才重新设置该能力的独立实施门。
