# ADR: 由 GitHub Action 构建并发布确定性生成物

Date: 2026-07-16  
Status: accepted

## Context

DCF Microcore 的连续版本和 DCF Next 首版都能够稳定发布完整 userscript。回顾实际提交链路后确认，大型生成物通常不是从对话沙箱经 GitHub 连接器直接搬运，而是先把本次修改涉及的源码、测试和构建规则提交到仓库，再由 GitHub runner 检出完整源码树、运行确定性构建，并使用仓库内的临时凭据提交生成物。

后续为了避免 GitHub Action 演变成远程编辑、调试和诊断系统，流程一度收紧为“本地修改与构建，再把全部结果一次性上传；Action 只读验证”。这个调整正确解决了远程往返、临时日志和碎片提交不断累积的问题，但同时取消了原本承担大型生成物发布的仓库内构建通道。当前对话环境没有稳定的本地 Git 凭据或文件直传桥，继续手工搬运完整 userscript 会把发布问题错误地变成编码、分片和参数传输问题。

GitHub Action 是否成为审判层，不取决于它是否写入仓库，而取决于它是否拥有业务判断和源码修改权。机械构建、固定校验与受限生成物提交可以由 Action 执行，不等于让 Action 决定需求、修复逻辑或评价架构。

## Decision

- 业务需求理解、源码修改、测试设计、版本决定和文档判断由当前维护者在本地或对话沙箱中完成，并以普通可审查文本提交到 GitHub。
- `dcf-chatgpt-next.user.js` 与 `dcf-chatgpt-next.meta.js` 被视为 `src-next/`、版本常量和构建脚本的确定性生成投影，由 GitHub Action 在仓库内部构建和发布。
- 发布 Action 可以：检出目标分支、运行既有构建、运行测试和语法检查、确认生成差异只涉及获准生成物，并在有变化时提交这两个生成文件。
- 发布 Action 不可以：应用业务补丁、修改 `src-next/`、修改测试或 ADR、根据失败日志自行修复、写入诊断文件，或把格式通过当作架构与实现正确的证明。
- Action 失败时，维护者读取证据、重新判断并显式修改源码；不在 workflow 中建立自动修复循环。
- workflow 必须避免机器人提交再次触发写入循环，并把写权限限制在执行生成物提交所需的最小范围。
- 本地构建仍可用于开发期快速反馈，但不再承担跨系统上传完整 userscript 的职责；审查分支上的正式生成物以仓库内确定性构建结果为准。

## Consequences

- 大型 userscript 不再经过模型输出、长工具参数、Base64 搬运或人工拼接，而在已经拥有完整源码树的 GitHub runner 内生成。
- 每次业务迭代仍可先在连续工作区完成和测试，避免恢复旧式“每次试错都提交、运行 Action、再根据远端日志继续修改”的高往返开发循环。
- Action 重新获得有限写权限，但写入对象被收缩为两个明确生成物；生成物之外的任何变化都必须使发布失败。
- CI 继续提供机械证据，不承担需求判断、架构裁决或自动修复责任。
- 发布历史可能包含一笔维护者源码提交和一笔机器人生成物提交；这是显式的源码—投影关系，不再强求把所有阶段压成一次本地全量上传。

## Reconsideration condition

当维护环境具备稳定、受认证且可直接传输完整文件的 Git 工作区时，可以改为由同一工作区提交源码与生成物；但仍必须保留确定性构建校验，并继续禁止 Action 或其他自动化自行修改业务源码。
