# ADR: 自动化只做维护辅助，不把格式合规当作正确性

Date: 2026-07-14  
Status: proposed

## Context

DCF 当前的正式性分散在源码、生成物、测试、README、architecture-current、current-state、maintenance skill、ADR 和 PR 说明中。大量内容由 AI 重复转述和同步，形成与功能修改无关的固定成本。

此前设想把这些要求整理成可执行的“合规”流程，但这仍然存在根本风险：当一次修改无法满足格式、清单或文档同步要求时，背后往往不是单纯的格式遗漏，而可能是修改范围失控、架构理解错误、事实源重复、测试对象错误，或者整条实现路线已经偏离。若工具只是驳回并要求重新满足格式，反而会掩盖根因，并给维护者制造“已经合规，所以路线正确”的虚假确定感。

因此，格式完整、测试通过、生成物一致和文档同步都只能作为证据，不能成为系统正确、架构合理或修改路线成立的替代判断。

## Proposed decision

- DCF 不建立独立的合规平台、自治审核器或格式审批门。
- 维护这次修改的 AI 统一负责理解需求、判断路线、控制范围、审查实现、解释证据并决定是否提交。
- 自动化只承担机械工作：生成派生产物、执行命令、收集事实、整理差异和减少重复写作。
- 自动化发现缺失、漂移、测试失败或生成不一致时，只暴露现象和上下文，不把“重新满足格式”预设为解决办法。
- AI 必须先判断异常是否说明更深问题：权威事实是否重复、实现是否走错层、抽象是否过度、测试是否验证错对象、文档是否承担了不该承担的职责。
- 测试通过只证明对应断言成立；构建一致只证明生成过程确定；文档字段一致只证明转录没有漂移。它们都不能单独证明功能、架构和用户体验正确。
- 文档收缩的目的不是更容易达到统一格式，而是减少平行事实源，使 AI 能直接看到系统真实结构和当前问题。
- ADR 只记录真正改变长期判断的决策。普通修改不因缺少某种形式文件而被视为不正规。

## Tool role

仓库脚本或 GitHub Actions 可以：

- 从源码生成 userscript、meta、Catalog、hash 和 package JSON；
- 执行 AI 选择的测试、构建和语法检查；
- 汇总 diff、文件变化、测试输出和生成物变化；
- 更新明确标记为派生事实的文档块与 ADR 索引；
- 把异常和证据返回给 AI。

它们不可以：

- 自行判断修改等级或架构是否合理；
- 以格式、manifest 或文档字段决定修改是否可以接受；
- 把测试通过提升为整体正确；
- 因为生成物一致就证明修改路线正确；
- 通过不断增加规则和例外，建立第二套长期治理系统。

## Intended workflow

```text
用户提出修改
→ AI 理解问题与当前系统
→ AI 选择实现路线并完成最小修改
→ 工具生成派生产物、运行命令并返回证据
→ AI 把异常视为调查入口，而不是格式修复任务
→ AI 回到源码、架构与用户现场判断根因
→ AI 修正、确认或放弃路线
→ AI 决定提交 / 合并
```

## Document contraction

- `README.md`：产品入口、安装/构建方式和稳定边界；
- `architecture-current.md`：一份当前结构解释；
- `current-state.md`：当前未完成事项、浏览器现场和下一步；
- `dcf-maintenance-skill.md`：维护动作和调查方法；
- `dcf-basic-consensus-prompt.md`：价值目标和少量不可违背边界；
- ADR：真实长期决策；
- 版本、包、模块、测试、生成物和索引等纯事实可自动派生。

README、architecture-current、current-state、basic consensus 和 maintenance skill 不再重复保存同一架构事实。机器生成块只是减少转录，不是新的正确性来源。

## Reconsideration condition

只要自动化开始让维护者把“通过检查”误认为“路线正确”，或者异常被简化成重新满足格式，就应当删除相应检查或降级为纯信息输出。成功标准是帮助 AI更早看见真实问题并减少机械劳动，而不是让仓库呈现出更完整的合规外观。
