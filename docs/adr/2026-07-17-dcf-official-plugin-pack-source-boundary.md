# ADR: DCF 官方插件工具包源码与交付边界

Date: 2026-07-17  
Status: accepted

## Context

DCF Next 目前由最小生存盒和十个普通插件共同组成。完整审查 userscript 将全部插件静态构建在一起，因此“源码位于哪个目录”“属于哪个插件包”“当前构建是否包含”曾被混为一件事。Local Agent 又包含浏览器代码和本机 Bridge 资源，容易被误拆成独立一级系统。

后续需要独立发布、选择安装和更新插件，但真实动态加载能力必须留到浏览器现场验收，不在无证据时提前改造生存盒。

## Decision

1. 建立 `plugin-packs/official/pack.json`，它是官方插件工具包成员、顺序、版本、入口、附带资源和推荐组合的唯一声明。
2. 插件归属由包清单决定，不由物理目录决定。当前插件源码继续保留在 `src-next/plugins/`，避免为了目录外观修改相对引用或建立兼容桥；每个插件仍只有一份正式源码。
3. Local Agent 的浏览器插件、协议模块、本机 Bridge、配置示例和说明均属于 `dcf.next.local-agent` 这一枚插件。Bridge 文件可以继续位于 `bridge/`，但包清单明确记录其资源归属。
4. 完整审查 userscript 不再扫描并自动吸收全部 `src-next/**/*.js`。构建器只装入生存核源码和官方包清单明确列出的模块。
5. 同一包清单同时生成独立的 `dist/dcf-official-plugin-pack.json`。工件包含清单、精确文件内容和 SHA-256，不包含生成时间，保证确定性。
6. 本次不改变浏览器运行方式。静态注册表、当前清单协调和完整 userscript继续作为捆绑兼容形态。
7. “新构建插件自动加入旧启动清单”明确属于捆绑兼容逻辑，不代表未来安装语义。动态加载切换成功后必须删除。
8. 本机代码库、精确运行快照与动态执行直接进入下一次真实浏览器验收；验收失败则保留当前捆绑运行形态，不增加多套 fallback 强行推进。

## Consequences

- 官方插件包已经成为独立、可审查、可确定性发布的交付物，同时不产生第二份插件源码。
- Local Agent 与其他插件处于同一包成员地位，本机 Bridge 只是其附带资源。
- 新插件只有写入官方包清单才会进入捆绑构建，构建器不再把目录扫描结果当作产品决定。
- 代码供应边界先成立，运行时安装与动态加载的真实风险被隔离到浏览器验收。

## Reconsideration conditions

- 真实动态加载验收要求插件源码采用不同的封装格式；
- 官方包需要同时保存多个插件版本，现有单路径入口不足；
- 第二个真实插件包出现，需要把当前官方包专用构建逻辑提炼成通用工具。
